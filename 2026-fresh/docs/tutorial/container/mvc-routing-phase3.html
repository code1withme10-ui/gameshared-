Phase 3 – Introduce Controller

Objective

We will introduce a Controller class to handle message-related logic. This keeps index.php clean and centralizes behavior.

Router remains unchanged. Only logic moves into Controller.

What Must Change / Key Concepts

  • Create /app/core/Controller.php base class
  • Create /app/controllers/MessageController.php
  • Move message form and listing logic from index.php to controller
  • Router will call controller methods instead of inline logic

Example base Controller:

<?php class Controller { // Render a view file with optional data protected function view($path, $data = []) { extract($data); require __DIR__ . "/../views/$path.php"; } }
Base class simplifies future controllers.

What to Check / Verification Steps

  1. Start server:
php -S localhost:8000 -t /var/www/tutorial/containers/public/
  1. Visit:
http://localhost:8000/messages/create
  1. Confirm sender form appears
  1. Visit:
http://localhost:8000/messages
  1. Confirm message listing text appears
  1. Visit unknown route:
http://localhost:8000/unknown
  1. Confirm "Page not found." appears
Router + Controller integration is working if all checks pass.

Terminology

  • Controller – Handles requests and business logic
  • Base Controller – Provides shared methods like view rendering
  • Dispatch – Router calls controller methods instead of inline code

Reflection

How does moving logic to Controller improve maintainability and clarity compared to Phase 2?

Cleaner index.php, reusable methods, prepares for Models in next phase.
