Phase T – Centralize Web Entry (Bridge Phase)

Objective

In previous lessons, you accessed:

http://localhost:8000/sender.php

In this phase, we will begin preparing for MVC by ensuring all web traffic passes through:

public/index.php
This is a bridge phase. We are not introducing MVC yet.

What Must Change / Key Concepts

  • We stop accessing sender.php directly.
  • index.php becomes the single entry point.
  • index.php decides what to load.

New access pattern:

http://localhost:8000/messages/create
Think of index.php as a traffic controller.
Do NOT delete sender.php.

AI Prompt / Guidance for Learner

Before modifying public/index.php, paste its entire contents into your AI session.

Then instruct AI:

This is Phase T – Centralize Web Entry. Please modify my public/index.php safely so that: - When URL is /messages/create - It loads sender.php - Without breaking existing functionality - Without introducing Router or Controller classes

Ensure existing behavior continues to work.

What to Check / Verification Steps

  1. Start server:
php -S localhost:8000 -t /var/www/tutorial/containers/public/
  1. Visit:
http://localhost:8000/messages/create
  1. Confirm sender form appears.
  2. Confirm no previous functionality is broken.
If everything works, you have successfully centralized web entry.

Terminology

  • Entry Point – The first file that receives a web request.
  • URI – The path part of a URL.
  • Manual Routing – Loading files based on the URI.

Reflection

Why is it better to have one entry file instead of many directly accessible PHP files?

This prepares your thinking for routing and MVC.
